This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A small toolkit that analyzes per-gene expression distributions from a *C. elegans* natural-isolate dataset (`worm.csv`). It does two independent things, both per gene:
- fits a **4-parameter Gaussian** + computes **truncation-index** metrics → `worm_fourparam_table.csv`;
- detects **KDE density peaks** (number / location / height / prominence) → `peaks.json`.

## The pipeline (requires reading 3 files to see end-to-end)

```
worm.csv                              bhuvanfitter.py
  (strain col + 207 isolate cols,       (BhuvanFitter.fit)
   25,849 gene rows)                           │
        │  set_index('strain').T               │
        ▼  → 207 strains × 25,849 genes         ▼
  generate_fourparam_stats.py ── per gene ──► worm_fourparam_table.csv
        │   (build_table loops df.columns)      (25,849 rows, 19 cols)
        └── commits + pushes the CSV to origin
```

- Genes with `< 10` finite observations, or whose `curve_fit` fails to converge, are written as a row with `fit_success=False` and NaN metrics rather than skipped or crashing.
- `generate_fourparam_stats.py` has a module-level `COLUMNS` list that **must stay identical to the keys `BhuvanFitter.fit("fourparam")` returns** (same names, same order). If you add/rename a key in the fit dict, update `COLUMNS`.
- `load_expression(csv_path, id_col="strain", drop_cols=())` is the shared loader (genes as rows → transposed to samples × genes). It is **dataset-agnostic**: `id_col` names the gene-identifier column (the worm file's first column is mislabeled `strain` but holds gene names; GTEx uses `Name`) and `drop_cols` discards extra non-sample index columns (e.g. GTEx's `Description`). Defaults reproduce the original worm behaviour exactly.

A near-identical variant, `generate_fourparam_stats_excluded.py`, fits the same 4-parameter Gaussian but **drops every expression value `<= EXCLUDE_AT_OR_BELOW`** (default `-1`) from each gene's array before fitting (so `n_obs` shrinks and a gene can fall below `MIN_OBS`). It imports `COLUMNS`, `MIN_OBS`, `load_expression`, `git_push`, and `_failed_row` from `generate_fourparam_stats.py` (single source of truth — only `build_table` / the parallel worker, the threshold constant `EXCLUDE_AT_OR_BELOW`, and output naming differ). **The output filename encodes the input dataset and the threshold** via `output_csv_for(threshold, input_path)` — `<input stem>_fourparam_table_excluded_at_or_below_<threshold>.csv` — so datasets never collide: worm → `worm_fourparam_table_excluded_at_or_below_-1.csv`, GTEx → `cerebellumlog2_fourparam_table_excluded_at_or_below_-1.csv`. The earlier worm `-0.75` run is preserved as `worm_fourparam_table_excluded_at_or_below_-0.75.csv`.

  - **Flags:** `--threshold T` (exclusion cutoff, default `-1`), `--limit N`, `--no-push`, plus `--input PATH` / `--id-col COL` / `--drop-col COL` (repeatable) for targeting other datasets through the generalized `load_expression`, and **`--jobs N`** (parallel worker processes; genes are independent so this scales near-linearly) and **`--max-nfev N`** (curve_fit cap, forwarded to `fit("fourparam")`). Parallelism uses a `multiprocessing.Pool` whose workers cache the threshold / `max_nfev` via `_init_worker`; the module-level `_fit_one((gene, values))` worker is picklable and `imap` preserves input order, so a parallel run is **bit-identical** to `--jobs 1`. The GTEx cerebellum table was generated with `--input cerebellumlog2.csv --id-col Name --drop-col Description --threshold -1 --jobs 11 --max-nfev 2000`.

A parallel pipeline produces the peak dictionary:

```
worm.csv ──► generate_peaks.py ──► peaks.json
   (load_expression, same loader)   (gene_peaks per col)   (pushed to origin)
```

- `peaks.json` is a nested dict `{gene: {peak_expression_value: {"height":..., "prominence":...}}}`. Peak count per gene = `len()` of the inner dict; a gene with no detectable mode maps to `{}`. **JSON keys are strings**, so the peak expression values are stored as strings — parse to float on load.
- `generate_peaks.py` reuses `load_expression` and `git_push` imported from `generate_fourparam_stats.py` (single definition of each — `git_push` is file-agnostic).

## `BhuvanFitter` contract

- Two fit models are registered in `_FIT_REGISTRY` and dispatched through
  `fit(model, **kwargs)`: **`"fourparam"`** and **`"kde"`**. `active_fits` tracks
  which have run; each fit sets its flag, and `hist(lines=[...])` only draws fits
  that have been run. `**kwargs` are forwarded to the chosen fit method
  (`fourparam` takes `max_nfev`; `kde` takes the `gene_peaks` knobs).
- Histogram is **always 40 bins** (`BhuvanFitter.BINS`).

**`fit("fourparam")`**
- The fit uses `curve_fit(..., method="trf")` with **default linear loss = ordinary least squares**. This is deliberate: it genuinely minimizes the residual sum of squares. (An older Colab version used `loss="soft_l1"`, which is robust but does *not* minimize SSE — it yields different `x0/w` and therefore different metrics. Do not switch back unless intentionally matching that legacy output.)
- `x_max` (constructor arg, defaults to the data max) is the truncation ceiling used by the metrics.
- `fit("fourparam", max_nfev=...)` caps the curve_fit function evaluations (default `10_000`). Genuine fits converge well under it; lowering it (e.g. `2000`) mainly cuts wasted effort on non-converging genes — which otherwise burn the whole budget before raising — at the small risk of flipping a slow-converging gene to `fit_success=False`. The generators expose it via `--max-nfev`.
- Returns a 19-key dict:
  `gene, y0, A, x0, w, sumsquarevalue, ti_fourparam_sigma_dist, truncationindex, min, max, mean, std, skew, kurt, right, maxheight, rightheight, n_obs, fit_success`
- `mean`, `std`, `skew`, and `kurt` are pure per-gene summary statistics of the finite values used for the fit (`mean = data.mean()`, `std` = **sample** standard deviation, `ddof=1`; `skew`/`kurt` = `scipy.stats.skew`/`kurtosis` with defaults, i.e. `bias=True` and **Fisher excess** kurtosis so a normal distribution is 0 — matching the notebook's `build_shape_features`; all NaN when `n_obs <= 1`, and `skew`/`kurt` are additionally NaN for a **constant** array even when the fit "succeeds"). For the excluded variant they reflect the post-exclusion array. They sit alongside `min`/`max`/`n_obs` and do not depend on whether the fit converged (but a `fit_success=False` row still writes them as NaN, like every other metric). `skew`/`kurt` were added after the tables were first generated; the **cerebellum table (`cerebellumlog2_fourparam_table_excluded_at_or_below_-1.csv`) was backfilled** with these two columns (computed on the same `> -1` array, bit-identical to a regen), while the worm tables gain them only when next regenerated.
- `truncationindex` (the renamed height-ratio metric) `== rightheight / maxheight`, where the baseline subtracted from both is the **fitted curve's minimum over the histogram interval** (`_curve_baseline()`, i.e. `min(f)` on a 600-pt grid over `[hist_edges[0], hist_edges[-1]]`) — **not** `f(data min)`. So `maxheight = max(f) − min(f)` and `rightheight = f(x_max) − min(f)`. Because the baseline is the curve's true interval-minimum, the ratio is **bounded to [0, 1]** (0 = ceiling at the curve minimum, 1 = ceiling at the peak). It returns **NaN** only when `maxheight == 0` (the curve is flat over the interval). (An earlier version used `f(data min)` as the baseline, which left the ratio unbounded — values ranged to ±1e14 — so don't reintroduce that.)
- Metric properties (`truncationindex`, `ti_fourparam_sigma_dist`, `maxheight`, `rightheight`) raise `RuntimeError` until `fit("fourparam")` has been called.

**`fit("kde")`**
- Runs a Gaussian KDE and **reuses the module-level `gene_peaks`** for mode
  detection, so the peaks it reports are identical to `generate_peaks.py` /
  `peaks.json`. Accepts `bw_method`, `min_prominence_frac`, `grid_size`,
  `pad_frac` (same meanings as `gene_peaks`).
- Returns a 6-key dict:
  `gene, n_peaks, peaks, bw_method, n_obs, fit_success`, where `peaks` is the
  `gene_peaks` dict `{value: {"height", "prominence"}}` and `n_peaks == len(peaks)`.
- `fit_success` is `False` (and the cached density is `None`) only when the KDE is
  singular; peak detection still returns `{}` rather than raising. `kde_function(x)`
  evaluates the cached KDE (raises `RuntimeError` if not run / didn't converge).

**Plotting** — `hist(lines=["fourparam", "kde"])` overlays either/both fitted
curves on the 40-bin histogram. The KDE density (which integrates to 1) is scaled
onto the bin-count axis by `n_obs * bin_width`, with `▼` markers at each detected
peak. Unrecognised / not-yet-run / non-converged fits are skipped with a warning.

## `gene_peaks` contract

`gene_peaks(values, min_prominence_frac=0.05, bw_method="silverman", grid_size=1000, pad_frac=0.05, round_to=6)` returns `{peak_expression_value: {"height": <kde density>, "prominence": <prominence>}}` for one gene.

- Detection is on a **Gaussian KDE** (bin-independent), keeping `find_peaks` modes with prominence ≥ `min_prominence_frac` of the max density. Defaults reuse the former `find_density_peaks` settings; tune via the args.
- Returns `{}` for degenerate input (`< 5` finite points, no spread, singular KDE, or no interior mode) — never assume a gene has peaks.
- Peak-value keys are rounded to `round_to` decimals.

## Commands

There is no build system, linter, or test suite. Development is `python` scripts + the notebook. Requires `numpy`, `pandas`, `scipy`, `matplotlib` (plus `openpyxl` if you read `Supplementary Data 1 trunc 20250702.xlsx` via `pd.read_excel`, and `scikit-learn` for `cerebellumbhuvanfitter.ipynb` section 7's pTriplo-prediction model).

```bash
# Regenerate the full table from the CSV and push it to the repo (~25.8k genes)
python generate_fourparam_stats.py

# Quick sanity check: first 50 genes, write the CSV but DON'T push
python generate_fourparam_stats.py --limit 50 --no-push

# Generate the full table locally without pushing
python generate_fourparam_stats.py --no-push

# Excluded variant: default threshold -1, or override per run (filename encodes it)
python generate_fourparam_stats_excluded.py                    # -> worm_..._at_or_below_-1.csv (push)
python generate_fourparam_stats_excluded.py --threshold -0.75  # -> worm_..._at_or_below_-0.75.csv
python generate_fourparam_stats_excluded.py --limit 50 --no-push

# Excluded variant on another dataset (GTEx cerebellum), parallel + lower curve_fit cap
python generate_fourparam_stats_excluded.py --input cerebellumlog2.csv \
    --id-col Name --drop-col Description --threshold -1 --jobs 11 --max-nfev 2000 --no-push
    # -> cerebellumlog2_fourparam_table_excluded_at_or_below_-1.csv

# Peak dictionary — same flags (all genes + push / sanity subset / local only)
python generate_peaks.py
python generate_peaks.py --limit 50 --no-push
python generate_peaks.py --no-push
```

All generators take `--limit N` and `--no-push`; `generate_fourparam_stats_excluded.py` additionally takes `--threshold T` (default `-1`, sets the exclusion cutoff **and** the output filename), `--input` / `--id-col` / `--drop-col` (target another dataset), and `--jobs N` / `--max-nfev N` (parallelism and the curve_fit cap). After a `--limit` run the output file holds only those genes; restore the committed full version with `git checkout -- <file>` before pushing anything. To regenerate several thresholds at once, run each with `--no-push` (distinct output files, no git race) and make one commit.

There are **two parallel notebooks**, one per dataset, with identical structure — they differ only in the data sources they load (and the example gene):
- **`wormbhuvanfitter.ipynb`** — the worm dataset (`worm.csv` → `master`, `worm_fourparam_table_excluded_at_or_below_-1.csv` → `fourparam_df`). This is the original notebook (renamed from `newbhuvanfitter.ipynb`; the worm CSVs were renamed from `Supplementary Data 1_csv.csv` / `fourparam_table*`).
- **`cerebellumbhuvanfitter.ipynb`** — the GTEx cerebellum dataset (`cerebellumlog2.csv`, loaded with `.drop(columns=['Description']).set_index('Name').T`, → `master`; `cerebellumlog2_fourparam_table_excluded_at_or_below_-1.csv` → `fourparam_df`).

Both are the interactive scratch space (`from bhuvanfitter import BhuvanFitter, gene_peaks`) and do three things. When you change one notebook's shared analysis structure, mirror it in the other:
- **Per-gene inspection** — build a `BhuvanFitter` on one column of `master` (the transposed source CSV) and view its `fit("fourparam")` / `fit("kde")` and the `hist(lines=["fourparam", "kde"])` overlay.
- **Cross-gene distribution plots** — it loads that dataset's excluded fourparam table into `fourparam_df` and defines two helpers for histogramming any column across genes: `select(param)` applies the **single shared filter** (`fit_success == True`, `0 < truncationindex < 1`, and `n_obs >= MIN_OBS`, NaNs dropped) and `plot_param_hist(param, *, color, bins, log)` draws the histogram (returning the data) and prints a filtering funnel — gene counts at each stage (master total → fourparam_df total → fit_success → NaN drop → truncationindex > 0 → < 1 → n_obs >= MIN_OBS). Change the filter in one place (`select`) and both the sumsquarevalue and truncationindex plots follow. **`MIN_OBS` is a notebook-level constant (default 30)** applied as an analysis-time floor on the table's `n_obs` column — the generator's own `MIN_OBS=10` is too low for a 4-param fit to a 40-bin histogram, and filtering here is identical to regenerating the table with a higher floor (a gene's fit is independent of `MIN_OBS` once `n_obs` clears it), so no regeneration is needed to tune it.
- **Per-gene expression summaries (filtered)** — `plot_gene_stat_hist(stat, source='full', *, color, bins)` is a third shared helper (alongside `select` / `plot_param_hist`) that histograms, over the **same filtered genes** (`select('gene')`), a per-gene **mean** or **std** of expression. `source='full'` reduces `master` over **all** samples (incl. the −1 floor); `source='table'` instead plots the post-exclusion `fourparam_df[stat]` column (only `> −1` samples) — genuinely different numbers, so pair the mean/std calls with the **same `source`** if feeding the scatter. The last three cells are now `strain_avgs = plot_gene_stat_hist('mean')`, `strain_sds = plot_gene_stat_hist('std')`, plus the unchanged **mean-vs-SD scatter** (`strain_avgs` x, `strain_sds` y; aligned by gene index). Run them in order (the scatter reuses `strain_avgs` / `strain_sds`).
- **Genes-of-interest truncation ranges (worm only)** — `wormbhuvanfitter.ipynb` has a final section that loads `genes_of_interest.json` (keys `mco_dev` / `mco_behavior` / `lof_dev` / `lof_behavior`, each a gene→transcript-ID map whose IDs match the table's `gene` column), builds **7 categories** (the 4 keys + deduped-union `mco_combined` / `lof_combined` / `all_combined`), and reports per-category `truncationindex` summaries (`n_requested`, `n_in_table`, `n_used`, min/max/mean/median) via `ti_ranges(apply_filter)` in **two cells**: one without the shared filter (any successfully-fit transcript) and one with it. Worm-specific (the transcript IDs are worm) — no cerebellum counterpart.
- **pTriplo vs. fourparam features (cerebellum only)** — `cerebellumbhuvanfitter.ipynb` has a final section ("5. pTriplo vs. the fourparam features") that correlates **pTriplo** (Collins 2022 triplosensitivity score, `dosage_sensitivity_collins2022.tsv.gz`; "high" cutoff ≥ 0.94) against **every numeric fourparam feature**. pTriplo and the truncation index are both *continuous classifier scores* for triplosensitivity (neither is ground truth); this is the descriptive agreement view. The fourparam table is keyed by Ensembl IDs and pTriplo by gene symbols, so the join uses the GTEx source's own two id columns (`Name` = ENSG, `Description` = symbol — version-stripped) as the ENSG→symbol map; no external lookup. Restricted to `fit_success == True` (≈15.5k genes joined), it prints `ptriplo_corr` (Spearman ρ + Pearson r + n per feature, sorted by |ρ|) and a `pTriplo` vs. `feat` scatter (default `feat="truncationindex"`). Cerebellum-specific (pTriplo is human; the worm table's transcript IDs would need ortholog mapping) — no worm counterpart. **Result: `truncationindex` is ~uncorrelated with pTriplo (ρ ≈ −0.01); the expression-level features (`mean`/`min`/`x0`/`max`, ρ ≈ 0.43) carry the signal.**
- **Truncation index conditioned on fit quality (cerebellum only)** — section "6. Truncation index vs dosage scores, conditioned on fit quality" tests whether `truncationindex` only tracks dosage sensitivity for genes the Gaussian actually fits. Fit quality is **R² = 1 − sumsquarevalue/SST** (scale-free), *not* raw `sumsquarevalue` (which is entangled with n_obs/std/A and would bias the correlation). **No refit** — `y0,A,x0,w` and `sumsquarevalue` come from the table; each gene is only re-histogrammed via `BhuvanFitter(vals>−1, gene)` (the constructor caches `hist_counts`; no `curve_fit`) to get `SST = Σ(counts − mean)²`. Builds `cond` (the `select()`-filtered genes inner-joined to **both `pTriplo` and `pHaplo`** + the `r2` column), then (a) a **stratified trend** — Spearman ρ(`truncationindex`, score) within R² deciles, to see if |ρ| rises with fit quality — and (b) `cond_corr`, the full feature-vs-pTriplo/pHaplo correlation table on the well-fit subset (`r2 >= R2_MIN`, default 0.9), plus a pTriplo-vs-truncationindex scatter colored by R². **Result: the hypothesis is NOT supported** — ρ(`truncationindex`, pTriplo/pHaplo) stays ≈ 0 across all R² deciles and on the well-fit subset (≈0.03 / 0.07 at R² ≥ 0.9), while expression-level features keep tracking the scores. Cerebellum-specific.
- **Predicting pTriplo from distribution features (cerebellum only)** — section "7. Predicting pTriplo from distribution features" asks the multivariate question: can a model reconstruct pTriplo from a gene's distribution **shape**? Two feature lists: `LEVEL` (mean/min/max/x0/std/n_obs/A) and scale-free `SHAPE` (skew, kurt, r2, cv=std/mean, w_x0=w/x0, truncationindex, ti_fourparam_sigma_dist). `build_shape_features(df)` adds these with **no refit** (skew/kurt from the post-exclusion values via `scipy.stats`, `r2` from `BhuvanFitter`'s cached histogram, ratios). It runs a **falsifiable EDA gate** (univariate ρ per feature + KMeans clustering of `SHAPE` → per-cluster pTriplo, a "target group" check), a **5-fold CV increment table** (`level`/`shape`/`combined` × Ridge/HistGBM; the deliverable is whether `combined` beats `level`), then fits the combined HistGBM, prints permutation importance, plots out-of-fold predicted-vs-actual, and **applies the model genome-wide** to every `fit_success` gene → `cerebellum_shape_ptriplo_pred.csv` (`gene, symbol, shape_pTriplo, pTriplo`; ~47.8k genes). `TARGET` is a cell-level constant (`"pTriplo"`; set to `"pHaplo"` for the identical pipeline). `n_peaks` was deliberately **not** used (KDE ~0.18 s/gene → infeasible genome-wide, and it added no signal). Requires `scikit-learn`. **Result: shape-only ≈ level-only (ρ ≈ 0.36 vs 0.38, R² ≈ 0.14 vs 0.15) and combined ≈ level — shape and level are redundant, not additive, with a low ceiling (R² ≈ 0.15)**; single-tissue distribution features reconstruct only a fraction of pTriplo. Cerebellum-specific (the trained model could later be applied to the worm table).

## The grant proposal (`ACFROG~1.PDF` → `acfrog.md`)

`ACFROG~1.PDF` is the Pierce-lab NIH-style grant proposal ("OE threshold / Censoring Hypothesis" for DS & AD genes) whose preliminary figures this whole toolkit supports. `acfrog.md` is its markdown conversion (text reconstructed from the PDF; Figs 1 & 2 are conceptual diagrams/tables and are only described — but Fig 2A's gene table is transcribed). **`regenerate_acfrog_figures.py`** regenerates the two *data* figures **faithfully from repo data** via `BhuvanFitter` → `acfrog_figure3_worm.png` (Fig 3, worm) and `acfrog_figure4_cerebellum.png` (Fig 4, cerebellum: Hsa21 & triplication-sensitive gene distributions). Run: `python regenerate_acfrog_figures.py`.

- **Fig 3 OE groups come from Figure 2A, not `genes_of_interest.json`.** The script hard-codes `SENS_GENES` (24 OE-sensitive) / `TOL_GENES` (23 OE-tolerant) worm gene names extracted from Fig 2A's mcOE-"Any" shaded cells (the json lacks the OE-tolerant no-phenotype genes, which are the correct "absent" control for panel D). `build_worm_groups()` resolves those names → fourparam-table transcript IDs (`w{n}_{transcript}`) via the `Supplementary Data 1 trunc 20250702.xlsx` `GeneName`/`transcript`/`wwww` columns (hlh-34, sod-5, H39E23.3 aren't in the dataset and drop out). **`EXCLUDE_GENES = {"itsn-1", "adr-2"}`** (two OE-tolerant genes) are dropped from `SENS_GENES`/`TOL_GENES` at the professor's instruction, so they're absent from every worm panel (3A–D and the 3B comparison), filtered and no-filter alike.
- **Honesty caveat (in script + md):** with the proper Fig 2A groups, Fig 3D shows the qualitative pattern as a **tail effect** — OE-tolerant genes are capped (max TI ≈ 0.30) while OE-sensitive spread to 0.60 — but the medians are similar so Mann-Whitney is only p ≈ 0.24 (n.s.), weaker than the proposal's p < 0.004 (likely the older *unbounded* TI metric amplified it). Cerebellum (Fig 4): triplication genes have truncation index ≈ 0, and even as a pTriplo≥0.94-vs-dup-tolerant contrast the signal is flat/reversed (consistent with the section-5 ρ ≈ −0.01). Fig 3D uses **probability (per-bin fraction) normalization** to match the original's `p` axis, not density.

## POS-vs-TOL dosage classifier (`train_dosage_classifier.py`)

A supervised 0-1 classifier that scores any cerebellum gene for dosage-sensitivity from its expression-distribution features, trained on the two curated human gene sets: **positives** = `positive_genes.txt` (44 OE/dosage-sensitive symbols, "POS", label 1), **negatives** = `positiveANDnegativeControlGenes.csv` col-0 (677 duplication-tolerant symbols, "TOL", label 0). Purpose: fit the model now, then extrapolate a predicted POS/TOL score to every other gene. Run: `python train_dosage_classifier.py` (`--no-genome` skips the genome-wide CSV; `--seed N`). Companion `ablation_dosage_features.py` reports which feature subsets carry the signal.

- **Data build (`load_labelled`).** Reads the cerebellum fourparam table (`cerebellumlog2_fourparam_table_excluded_at_or_below_-1.csv`), restricts to `fit_success == True`, maps ENSG→symbol via the GTEx `Name`/`Description` id columns (same bridge as notebook §5–7), labels by symbol membership (POS wins any overlap; there is none), and keeps **one canonical row per symbol** (the fit_success ENSG with the most `n_obs`) so there's no within-symbol leakage across CV folds. Coverage: **42 POS, 472 TOL** map to fit_success rows (~11:1 imbalance, prevalence 0.082).
- **Features (table-only, no refit/re-histogramming).** `BASE_FEATURES` = the numeric fourparam columns (`y0,A,x0,w,ti_fourparam_sigma_dist,truncationindex,min,max,mean,std,skew,kurt,maxheight,rightheight,n_obs`) + `DERIVED` scale-free ratios `cv=std/mean`, `w_x0=w/x0` (17 total). Median-imputed; standardized for the linear model.
- **Method.** Compares LogisticRegression (L2, scaled) / RandomForest / HistGradientBoosting — all `class_weight`-balanced — by **repeated stratified 5-fold×10 CV** on ROC-AUC & PR-AUC, tunes the winning family by `GridSearchCV`, then reports **out-of-fold** ROC/PR/confusion at an F1-optimal threshold, permutation importance, and (logreg) standardized coefficients. Emits `dosage_classifier_eval.png` (ROC + PR + top-12 importance), saves the fitted pipeline to `dosage_classifier.joblib` and metrics to `dosage_classifier_metrics.json`, and scores **all 48,888 fit_success genes** → `cerebellum_dosage_classifier_pred.csv` (`gene,symbol,dosage_score,label,n_obs`, sorted by score).
- **Why not "accuracy".** At 11:1 imbalance a trivial all-TOL model already scores ~0.92 accuracy, so accuracy is reported only at the F1-optimal threshold for completeness; **ROC-AUC and PR-AUC (vs the 0.082 prevalence baseline) are the honest headline metrics.**
- **Result.** Winner = **logistic regression, C=0.01** (heavy regularization): **CV/OOF ROC-AUC ≈ 0.82, PR-AUC ≈ 0.39–0.45** (≈5× the 0.082 baseline). **Expression *level* is the stronger signal**, but distribution *shape* alone, tested honestly, is also real. Ablation (`ablation_dosage_features.py`): `mean`-only ROC-AUC **0.824** ties/beats the full model (0.816), `top3_level` (mean/min/max) 0.820, `level_only` 0.818, and **`truncationindex`-only 0.528 (≈chance)**; adding shape to level does not help the combined model. So the classifier is essentially a "cerebellum mean-expression" detector — POS genes are more highly expressed in cerebellum than the TOL controls — which corroborates the repo's earlier finding that expression-level features (ρ≈0.43) carry the pTriplo signal while the truncation index does not (ρ≈−0.01). **Caveat:** this makes the score partly a tissue-expression confound (the genome-wide top hits are dominated by very-high-expression MT-\* genes), not evidence that distribution *shape* alone drives dosage sensitivity.
- **Shape-alone test (strict scale-invariance).** The first `shape_only` ablation (ROC-AUC 0.712) was contaminated — it included raw `w` (the Gaussian's absolute width), which scales with expression magnitude like `mean`/`min`/`max` and isn't actually shape. `PURE_SHAPE = [skew, kurt, cv, w_x0, truncationindex, ti_fourparam_sigma_dist]` (6 dimensionless ratios/moments, no absolute-unit column — invariant to rescaling a gene's values) is the honest shape-only set: at the fixed C=0.01 tuned for the 17-feature model it scores 0.709 (barely different, since heavy regularization already suppressed `w`), but **C=0.01 turns out to be badly over-regularized for a 6-feature set** — a dedicated C sweep shows performance rising monotonically as regularization weakens, plateauing at **C≥1: ROC-AUC ≈ 0.775, PR-AUC ≈ 0.30**. So distribution shape *alone*, independent of level, does carry real above-chance signal (~0.78 vs 0.5), just weaker than level alone (0.818) or mean alone (0.824) — level and shape both encode some of the POS/TOL split, level more strongly. `ti_fourparam_sigma_dist = (x_max−x0)/(w/√2)` is confirmed affine-invariant (both terms are lengths in the same expression units, so a rescale/shift of the data cancels in the ratio) — correctly a shape, not scale, feature.
- **Combining level + shape with per-group regularization (`train_grouped_regularization.py`).** Tests whether shape adds anything *on top of* level once each group is allowed its own optimal regularization strength, rather than one global `C` for all features. Plain `LogisticRegression` only exposes a single `C`; a `GroupScaler` transformer (in the pipeline, after `StandardScaler`) multiplies just the `PURE_SHAPE` columns by a tunable `scale` before the shared `C` is applied — scaling a group down forces a larger raw coefficient for the same effect, which the L2 penalty sees, so `scale < 1` tightens that group's *effective* penalty relative to level and `scale = 1` recovers the ordinary uniform-penalty combined model. Jointly grid-searching `(C, scale)` (same repeated 5×10 CV) explores the full space of relative level/shape regularization. **Result:** best combo (`C=0.003`, `shape_scale=0.03`, i.e. shape penalized ~1100× more than level) reaches **ROC-AUC 0.821 / PR-AUC 0.463**, barely above the uniform-penalty combined baseline (0.819/0.453, +0.002 ROC-AUC) and still below `mean`-only (0.824). The grid search's own optimum wants to nearly zero out shape's contribution — i.e. even given the freedom to weight the two groups completely independently, the best model essentially discards shape. **Conclusion: level and shape are redundant, not additive, for this task** — matches the repo's existing GTEx pTriplo finding (notebook §7: "shape-only ≈ level-only ... combined ≈ level"). Shape carries real signal in isolation (§ above) but none of it survives once level is already in the model.

## Direct test of the grant's Censoring Hypothesis (`evaluate_censoring_hypothesis.py`)

`grant.pdf` (the current NIH proposal) is itself **corrupted at the byte level** — every page's content stream fails zlib decompression in both `pypdf` and PyMuPDF, and a `pikepdf`/`qpdf` structural-repair pass can't recover the compressed bytes either (it's untracked in git, so this predates/is unrelated to this repo's `.gitattributes`/LFS setup). The claim was instead read from `acfrog.md`, the markdown conversion of the prior version of the same proposal (CLAUDE.md already treats them as "same project, same core hypothesis" — only specific gene-set figures were updated between versions). **The Censoring Hypothesis:** a gene's natural expression-level distribution across a healthy population should be Gaussian; if a gene has a deleterious OE threshold, individuals who'd express past it are effectively censored from the "healthy" sample, truncating the right tail (an expression "ceiling"). OE-sensitive genes (POS) should show a truncated/capped distribution; OE-tolerant genes (TOL) an uncapped one — tested in the proposal in both wild *C. elegans* and human GTEx tissue.

This script tests that claim more rigorously than the plain/grouped-regularization dosage classifier above, which only splits credit between correlated level/shape groups linearly (a split that can still let level leak through nonlinearly). Instead it **matches each POS gene to its `k=5` nearest TOL genes by mean expression** (`sklearn.neighbors.NearestNeighbors`, with replacement, dedup'd) — explicitly controlling for expression level — then asks whether shape/truncation still separates POS from TOL **within** that level-matched set, in **both species**:

- **Human (cerebellum):** reuses `train_dosage_classifier.load_labelled()` directly (POS=42, TOL=472).
- **Worm (new `load_labelled_worm()`):** bridges POS/TOL human symbols to worm orthologs via `regenerate_grant_figures.human_to_worm()` (reads the existing `human_worm_orthologs.tsv` cache, no new network fetches) and `build_worm_groups_names()`, restricted to `fit_success == True` in the worm excluded table, one canonical transcript per worm gene (POS=87, TOL=324). The worm table lacks `mean/std/skew/kurt` (only backfilled for cerebellum) — computed via a **no-refit** pass over `worm.csv`'s raw values for just the ~400 labelled genes (mirrors the cerebellum notebook's `build_shape_features`; confirmed the table's `gene` IDs, e.g. `w1_Y110A7A.10.1`, are literally `worm.csv`'s own gene-identifier values, so no id-mapping step is needed).
- Directional tests use the correct sign per metric: `truncationindex` (higher = more capped) tested `alternative="greater"`, `ti_fourparam_sigma_dist` (**lower** = ceiling closer to peak = more capped) tested `alternative="less"`; `skew`/`kurt`/`cv`/`w_x0` have no unambiguous a-priori direction from the hypothesis, so those get a two-sided test.

**Result — the hypothesis's shape signal does not survive level-matching, in either species:**

| | unmatched shape-only ROC-AUC | level-matched ROC-AUC | matched `truncationindex` p (greater) | matched `ti_fourparam_sigma_dist` p (less) |
|---|---|---|---|---|
| human cerebellum | 0.767 | **0.611** (OOF 0.582) | 0.56 (n.s.) | 0.33 (n.s.) |
| worm | 0.532 (already ~chance) | **0.476** (OOF 0.452, below chance) | 0.33 (n.s.) | 0.94 (n.s.) |

Neither of the hypothesis's own named truncation/ceiling metrics reaches significance in either species once TOL genes are chosen to match POS on expression level. (Human's matched set does show a secondary signal in `cv`/`w_x0`, two-sided p=0.001/0.004 — but those are level-adjacent width ratios with no clear a-priori direction from the hypothesis, not the specific truncation/ceiling mechanism it predicts, and matching on `mean` alone is imperfect at the tails — the highest-expressing POS genes still sit above their matched-TOL neighbors, per `censoring_hypothesis_eval.png`.) **Conclusion: once expression level is properly controlled for via matching, this repo's data does not support the Censoring Hypothesis's incremental claim that distribution shape/truncation predicts OE-sensitivity beyond level — in either species.** Outputs: `censoring_hypothesis_eval.png` (expression-level match diagnostic + matched ROC per species), `censoring_hypothesis_summary.csv`. Run: `python evaluate_censoring_hypothesis.py`.

## Why doesn't the grant's own claimed effect reproduce? (`evaluate_grant_original_genesets.py`)

The natural follow-up question: the evaluation above uses this repo's genome-wide POS (44-gene curated OMIM/G2P list) / TOL sets, which are **not** the exact gene groups the grant's preliminary figures were built on. To isolate whether the null result above comes from a different truncation-index formula or from different genes, this script replicates the grant's **own** stated gene definitions as closely as possible and tests them with this repo's `truncationindex` — nothing else changed.

- **Worm (Fig 3B): already tested, no new script needed.** `regenerate_acfrog_figures.py`'s `figure3b_comparison` uses the grant's *exact* Fig-2A gene set (24 OE-sensitive / 23 OE-tolerant Hsa21 orthologs, `itsn-1`/`adr-2` excluded per the professor), one canonical transcript per gene, and a mean±SEM/Welch-t-test — matching the grant's own representation as closely as this data allows. Result (already in CLAUDE.md above): Welch t-test **p≈0.11** (filtered) / **p≈0.08** (no-filter) — same genes, same transcript selection, same statistic, my truncation index — still nowhere near the grant's claimed **p<0.004**. - **Human (Fig 4 / Fig 10A): tested here, for the first time with the correct gene definition.** acfrog.md's preliminary-results text is explicit that the human evidence came from **"most of 1,400 genes predicted to be triplication sensitive (Collins et al., 2022)"** vs **"774 triplication-tolerant genes"** — not the 44-gene curated OMIM list used everywhere else in this repo's classifier/hypothesis-test work. `evaluate_grant_original_genesets.py` uses the actual Collins 2022 pTriplo score (`dosage_sensitivity_collins2022.tsv.gz`) at `pTriplo>=0.94` (this repo's own documented "high" cutoff) — **1,559 genes genome-wide, 1,461 with a valid cerebellum fit — already more genes than the grant's own "~1,400"**, so this isn't an underpowered comparison — against the existing duplication-tolerant list (677 requested, 427 with a valid fit; the grant's own count is 774, so a comparable list of the same type/vintage). **Result: the categorical claim does not reproduce, and if anything the direction is reversed.** Fraction of genes with any detectable ceiling (`truncationindex > 0`): POS=**0.359**, TOL=**0.393** — i.e. *fewer* of the triplication-sensitive genes show a ceiling than the tolerant controls, the opposite of "most of the 1,400 exhibit a ceiling, most of the 774 do not." Output: `grant_original_genesets_eval.png` (CDF comparison). Run: `python evaluate_grant_original_genesets.py`.

## Tissue-aware dosage-sensitivity model (testing the brain-expression hypothesis)

A follow-up hypothesis (the professor's): maybe OE-tolerant genes are tolerant specifically because they are **not prominently expressed in brain** relative to other tissues, while OE-sensitive genes are more **brain-specific** — an axis the cerebellum-only classifier above can't distinguish from plain expression magnitude, since it only ever looks at one tissue. Tested via genome-wide GTEx multi-tissue expression, folded into a tissue-aware extension of the dosage classifier.

- **Expanded positive gene list (`positive_genes_compiled.txt`).** Unions `positive_genes.txt` (44) with the human genes from `grant_genes.csv` rows where `mcOE_Any` is marked (27 unique symbols, worm OE-sensitive) — **70 unique symbols** (only 1 overlap: `SON`). Rows with no mcOE phenotype are **not** added to the negative side (worm OE-tolerance and human duplication-tolerance are different evidence types, kept separate per the user's direction); TOL stays the existing `positiveANDnegativeControlGenes.csv` list, unchanged. `train_dosage_classifier.load_labelled()` now takes an optional `pos_file` argument (default unchanged: `positive_genes.txt`) so this expanded list is a drop-in swap without touching the original classifier's behavior.

- **Tissue-aware classifier (`train_tissue_aware_classifier.py`) — ablation result.** Joins the expanded-POS cerebellum fourparam features (`LEVEL`/`PURE_SHAPE`, 68 POS matched of 70 + 472 TOL = 540 labelled genes, prevalence 0.126) with the 8 tissue features (`TISSUE = [brain_cerebellum, brain_mean, brain_max, nonbrain_mean, nonbrain_max, log2fc_brain_vs_nonbrain, tau, brain_is_top]`, the last a binary "is the top tissue a `Brain_*` region"). Ablation (repeated 5×10 stratified CV, logreg):

  | feature set | ROC-AUC | PR-AUC | n_feat |
  |---|---|---|---|
  | `tissue_only` | 0.813 | 0.412 | 8 |
  | `shape_only` | 0.802 | 0.400 | 6 |
  | `level_only` | 0.826 | 0.490 | 10 |
  | `level+shape` | 0.829 | 0.495 | 16 |
  | `level+shape+tissue` | **0.832** | **0.498** | 24 |

Confirmed via permutation importance on the tuned final model (logreg, C=0.3, OOF ROC-AUC=0.822, PR-AUC=0.429): `brain_cerebellum` ranks **3rd** (0.039, ahead of most shape features) but the genuinely *new*, non-cerebellum-redundant tissue signals (`nonbrain_max` 0.015, `brain_is_top` 0.0065, `tau` 0.0062, `log2fc_brain_vs_nonbrain` 0.0025) are individually modest — smaller than `n_obs`/`mean`/`brain_cerebellum`/`kurt`. **Verdict: the brain-specificity hypothesis has real support (esp. the categorical "brain is the top tissue" 4× difference above) but it's largely a restatement of/correlated with expression level already captured by the cerebellum-only data — it doesn't meaningfully outperform the existing level+shape model once that's already fit.** Outputs: `dosage_classifier_tissue_aware_eval.png` (ROC/PR/importance), `dosage_classifier_tissue_aware.joblib`, `dosage_classifier_tissue_aware_metrics.json`, and genome-wide scores `dosage_score_tissue_aware.csv` (34,986 genes with both a cerebellum fit and a tissue match). Top genome-wide hits include known brain/myelin genes (`MBP`, `GFAP`, `PLP1` — the last also a curated POS gene, Pelizaeus-Merzbacher disease) alongside MT-* genes, consistent with the model partly re-deriving "highly and brain-specifically expressed" rather than a clean orthogonal signal. Run: `python generate_tissue_specificity.py && python generate_gene_tissue_prominence.py && python train_tissue_aware_classifier.py`.
- **Shared top-tissue check.** Of the compiled POS/TOL genes (`gene_tissue_prominence.csv`), **testis** is the one tissue both groups meaningfully share as a top-expressed tissue — but lopsidedly: TOL 34.6% (224/648) vs POS 8.6% (6/70). ## Testing the professor's confound hypothesis with Enrichr (`enrichr_tissue_enrichment.py`)

**The critique:** everything above (mean cerebellum expression predicting POS/TOL; POS's 4× higher brain-top-tissue rate) could be circular rather than meaningful, if POS was curated mostly from brain-disease literature and TOL mostly from population duplication-tolerance studies that skew toward testis/immune-restricted gene families. In that case "predicts POS vs TOL" might just mean "predicts which curation pipeline a gene came from," saying nothing about real overexpression tolerance. This needed testing with a rigorous, independent tool (Enrichr's REST API, `maayanlab.cloud/Enrichr`) rather than this repo's own informal GTEx-median ratio counting — hypergeometric/Fisher's-exact enrichment against curated tissue-expression gene-set libraries, with proper adjusted p-values and odds ratios.

- **Method.** `submit_list`/`get_enrichment` wrap Enrichr's `/addList` and `/enrich` endpoints (shelled out via `curl` — Python's `requests` TLS handshake reliably times out against this host in this environment, `curl` doesn't). Queried 5 libraries covering every database category the professor named: `GTEx_Tissues_V8_2023` (511 terms), `ARCHS4_Tissues` (108), `Human_Gene_Atlas` (84 — the closest available match; **there is no library literally named "Human_Protein_Atlas" in Enrichr today**, noted explicitly rather than silently substituted), `Jensen_TISSUES` (1,842), `Allen_Brain_Atlas_up` (2,192). Smoke-tested first: `GAD1`/`SYN1`/`RBFOX3` correctly surface strong brain terms (rank ≤4, adj p≤0.001 in GTEx), `PCSK9` shows none — pipeline validated before the real run.
- **Four gene lists tested:** **POS** (70), **TOL** (677), and — the explicit follow-up, testing whether this repo's own core metric has the same problem — **ceiling** (cerebellum genes with `truncationindex > 0.1`, 586 genes; a meaningfully-sized "has a detectable ceiling" group, comparable to TOL) and **non_ceiling** (`truncationindex == 0`, "fully uncapped" — genome-wide this is 27,667 genes, the same over-represented pile `select()`/`valid()` normally exclude elsewhere in this repo; downsampled to a random 650 for a fair size-matched Enrichr comparison, seed=0).
- **Result: genuinely mixed, library-dependent — not a clean confirm or refute.**
  - **`Human_Gene_Atlas` and `ARCHS4_Tissues` (the two broader, classic tissue-atlas libraries) support the professor's suspicion for POS.** POS's top `Human_Gene_Atlas` term is `PrefrontalCortex`, rank 1, **adj p=0.020** (significant), odds ratio 37.5. POS's top `ARCHS4_Tissues` term is `BRAIN (BULK)`, rank 1 (though adj p=0.59, underpowered at n=70). TOL, by contrast, shows **no** brain signal in either — TOL's `ARCHS4_Tissues` top term is `SPERM` (adj p=4.8×10⁻⁸, extremely significant) with its best brain term at rank 39 and odds ratio ≈0.00002 (essentially absent), and TOL's `Human_Gene_Atlas` brain term (`Hypothalamus`) is rank 15, odds ratio 0.10 (depleted, not enriched). This directly corroborates the professor's suspicion and this repo's own earlier `gene_tissue_prominence.csv` finding (POS 27% brain-top vs TOL's 34.6% testis-top).
  - **`GTEx_Tissues_V8_2023` shows the opposite direction on raw significance** — TOL's best brain term (`Brain - Amygdala Female 60-69 Up`) is far more significant (adj p=3.5×10⁻⁷, odds ratio 143) than POS's (`Brain - Hippocampus Female 40-49 Up`, rank 4, adj p=0.30, odds ratio 18). This looks like a **list-size / statistical-power artifact rather than genuine biology**: TOL is 10× larger than POS (677 vs 70 genes), and this library has 511 narrow, demographic-stratified terms (tissue × sex × age-band) — a big list has far more power to hit significance on *some* one of 511 narrow bins by chance/partial overlap, without reflecting a broad, consistent brain lean (TOL's *other* four libraries all show either no brain signal or active depletion, and its overwhelming, highly-significant, consistent signal is testis/sperm-related). Raw adjusted p-values aren't safely comparable across 10×-different list sizes; the odds ratio is the fairer effect-size metric, and even there TOL's brain enrichment is a single narrow hit, not a pattern.
  - **The ceiling/non_ceiling follow-up echoes the same (weak) direction as POS/TOL.** `ceiling` shows modest, non-significant brain leans across libraries (best odds ratios 27–45, similar order of magnitude to POS) with its single strongest hit being `Brain - Caudate (Basal Ganglia)` in GTEx (rank 1, adj p=0.28, not significant at this size). `non_ceiling` shows **no brain signal at all** in `GTEx_Tissues_V8_2023` (no brain term anywhere in the top 50) and nearly-zero odds ratios elsewhere (0.000006–0.4) — the most "non-brain" of all four groups tested. So the truncation-index "ceiling" concept does track the same brain-vs-nonbrain axis as POS/TOL, directionally — but, like POS itself, doesn't reach formal significance at this list size in most libraries.
- **Honest verdict:** the professor's confound is **real and partially confirmed**, but not total — it's not that POS *is* simply "the brain genes" and TOL *is* simply "the non-brain genes" (the GTEx-library discrepancy shows that framing is too clean), but TOL is unambiguously, strongly, and consistently a testis/tissue-restricted-elsewhere list rather than a genuine "overexpression-tolerant" list in any tissue-neutral sense, while POS carries a real, corroborated (if modest and underpowered-at-n=70) brain lean in the two broadest tissue atlases. This means **the tissue-aware classifier's brain-specificity signal (and the plain cerebellum-mean-expression signal before it) should be read with real suspicion as substantially reflecting how the gene lists were curated, not a clean independent biological signal** — exactly the caution the professor raised. Outputs: 20 raw per-list-per-library CSVs (`enrichr_<list>_<library>.csv`), `enrichr_tissue_summary.csv`, `enrichr_tissue_comparison.png`. Run: `python enrichr_tissue_enrichment.py`.

## pTriplo's own training gene sets (Collins et al. 2022) — what they actually are, and why the exact lists aren't recoverable

Investigated after being asked what pTriplo uses as its positive/negative training data, since this repo's classifier work parallels that design (curate positive/negative genes → fit a model on gene-level features → score genome-wide).

- **The four training sets, per the paper's STAR Methods** ("Genes and gene sets," verified by downloading the full paper PDF directly since PMC/ScienceDirect only return truncated excerpts via fetch tools): **Haploinsufficient (HI, N=264)** — ≥3 of 4 criteria: gnomAD v2.1 PTV-constrained; confirmed AD disease via LoF/HI per ClinGen ("high confidence") or DECIPHER/DDG2P ("confirmed"); intolerant of GTEx v7 low-expressor outliers; no LoF deletions in 8,342 gnomAD v2.1 non-neuro controls. **Triplosensitive (TS, N=178, pTriplo's positive set)** — ≥3 of 4: gnomAD v2.1 missense-constrained; confirmed AD disease via triplosensitive/GoF/other mechanism per ClinGen (any confidence) or DECIPHER/DDG2P; intolerant of GTEx v7 high-expressor outliers; no whole-gene copy-gain duplications in the same control set. **Haplosufficient (HS, N=228)** and **Triploinsensitive (TI, N=150, pTriplo's negative set)** — all 4 of: gnomAD-tolerant; *zero* disease association in OMIM, ClinGen, or DECIPHER/DDG2P (stricter than "no dosage evidence" — no disease link at all); high rate of the relevant GTEx v7 expression-outlier direction; ≥1 documented deletion (HS) or duplication (TI) in the same non-neuro control set. These 328 genes calibrate an empirical-Bayes step + an 8-model ML ensemble over 145 gene features, applied genome-wide to all 18,641 autosomal genes (1,559 clear the pTriplo≥0.94 cutoff used elsewhere in this repo — far more than the 178-gene TS training set alone, since the score generalizes beyond its training genes).
- **The literal 4 gene-symbol lists were never published, confirmed by checking three independent sources:** (1) the journal's Table S5 supplementary Excel (`ars.els-cdn.com/.../mmc5.xlsx`) — only feature *descriptions*/counts, no gene-level membership; (2) the Zenodo data deposit (`zenodo.org/record/6347673`, `gene_features_matrix.tar.gz`) — 145 continuous model-input features per gene, no training-label column; (3) the paper's own code repo (`github.com/talkowski-lab/rCNV2`, `analysis/gene_scoring/gene_scoring_analysis.sh`) — this **does** contain the exact recipe (a chain of `fgrep`/`comm`-style set intersections over named component files like `gnomad.v2.1.1.lof_constrained.genes.list`, `ClinGen.hc_haploinsufficient.genes.list`, `DDG2P.hc_lof.genes.list`, `gnomad_sv.v2.1.nonneuro.no_lof_dels.genes.list`, etc.) but those component files and the final `gold_standard.*.genes.list` outputs were staged in the authors' private Google Cloud bucket at runtime, never committed to git or archived publicly (confirmed absent from the repo at both the default branch and the `v1.0` release tag).
- **What was exported instead (`clingen_haploinsufficient_sufficient_evidence.txt`, `clingen_triplosensitive_any_evidence.txt`, `clingen_no_evidence_haploinsufficiency.txt`, `clingen_no_evidence_triplosensitivity.txt`):** the **current, live ClinGen Gene Dosage Sensitivity curation** (`search.clinicalgenome.org/kb/gene-dosage/download`), filtered to the categories matching Collins' own ClinGen-based criterion — "Sufficient Evidence for Haploinsufficiency" (425 genes) and "Sufficient/Emerging/Little Evidence for Triplosensitivity" (15 genes) for the two positive sets, "No Evidence for Haploinsufficiency" (319) / "No Evidence for Triplosensitivity" (1,410) as partial negative-side proxies. **This is only one of Collins' ~3–4 criteria per set** (the gnomAD constraint, GTEx v7 outlier, and gnomAD-SV control-CNV criteria are not reconstructed), and the "no evidence" files are a partial proxy for the HS/TI "zero disease association anywhere" requirement (ClinGen dosage curation alone, not cross-referenced against OMIM/DECIPHER too — e.g. a gene tagged "Gene Associated with Autosomal Recessive Phenotype" in ClinGen would still fail Collins' actual no-disease-association bar but isn't excluded here). **Validated as a faithful current-day proxy:** the 15-gene triplosensitive "any evidence" count matches the paper's own cited pre-study ClinGen catalog size exactly ("Before this work, the ClinGen Genome Dosage Map included just 15 triplosensitive genes"); `APP` — one of the grant's own named triplication-disease examples — is in the exported triplosensitive list.
